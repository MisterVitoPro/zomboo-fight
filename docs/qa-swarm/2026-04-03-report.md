## P0 -- Critical

### P0-001: player2="none" string passed to Zombie.update() causes crash
**File:** `gameLib/main.py:126,169`
**Confidence:** confirmed
**Agents:** logic-correctness, error-handling, edge-case

When only 1 controller is connected (or keyboard mode), `self.player2` is set to the string `"none"` (line 126). This string is passed to `zombieSprites.update()` (line 169), which passes it to `Zombie.update()` as the `player2` parameter. The zombie code then checks `jCount == 2` before accessing `player2.rect`, but if `jCount` somehow equals 2 without player2 being initialized, it would crash. More critically, the string `"none"` is stored and could be accessed by other code paths expecting a Player object.

### P0-002: Spawn rate min/max go negative, causing randrange crash
**File:** `gameLib/main.py:56-57`
**Confidence:** confirmed
**Agents:** edge-case, logic-correctness, data-integrity

Every time a new zombie spawner appears, `zombieSMin -= 6` and `zombieSMax -= 6`. Starting at 60/90, after 10 spawner cycles `zombieSMin` hits 0, and after 11 it goes negative. `random.randrange(negative, negative)` raises a `ValueError`. The game will crash after ~10-15 minutes of play.

### P0-003: BigZombie can never spawn -- unreachable code
**File:** `gameLib/main.py:68-74`
**Confidence:** confirmed
**Agents:** logic-correctness, edge-case

`zNum = random.randrange(1, 12)` produces values 1-11. The BigZombie branch requires `zNum > 13`, which is impossible. BigZombie is dead code that can never execute.

### P0-004: Menu reloads background image from disk every frame
**File:** `gameLib/main.py:210`
**Confidence:** confirmed
**Agents:** performance, resource-mgmt

Inside the menu's `while keepGoing == 1` loop, `pygame.image.load(dataFiles.mainMenuIm)` is called every frame (30 FPS). This performs disk I/O 30 times per second for the same static image. The image is already loaded on line 203 before the loop.

### P0-005: donePlaying referenced before assignment in menu()
**File:** `gameLib/main.py:199-226`
**Confidence:** confirmed
**Agents:** error-handling, logic-correctness

If the menu loop exits without any event setting `donePlaying` (e.g., window manager closes the display), `return donePlaying` on line 226 raises `UnboundLocalError`. The variable is only assigned inside event handlers.

## P1 -- High

### P1-001: Sprite groups never cleared between games
**File:** `gameLib/sGroups.py`
**Confidence:** confirmed
**Agents:** resource-mgmt, data-integrity, architecture

When a game ends and the player returns to the menu, then starts a new game, all sprite groups in `sGroups` still contain sprites from the previous game. The `game.__init__()` method calls `makeSprites()` which adds new sprites but never empties existing groups. This causes ghosts from previous games, double-updates, and growing memory usage.

### P1-002: Module-level joystick/mixer init runs at import time
**File:** `gameLib/sprites.py:11-25`
**Confidence:** confirmed
**Agents:** architecture, resilience

`pygame.joystick.init()` and `pygame.mixer.init()` are called at module import time (line 11-12). If pygame hasn't been initialized yet or if hardware isn't ready, this can fail. Additionally, joystick state is frozen at import time and never re-evaluated.

### P1-003: Bare except in Levels.get_at() silently swallows all errors
**File:** `gameLib/Levels.py:36-39`
**Confidence:** confirmed
**Agents:** error-handling, resilience

```python
def get_at(self, dx, dy):
    try:
        return self.image.get_at((dx, dy))
    except:
        pass
```

This catches ALL exceptions (including KeyboardInterrupt, SystemExit, MemoryError) and returns `None`, which is then compared to `self.WALL` -- it will never match, silently skipping wall tiles on errors.

### P1-004: Zombie accesses player.oldxy2 which may not be initialized
**File:** `gameLib/sprites.py:401-436`, `gameLib/player.py:98-99`
**Confidence:** confirmed
**Agents:** error-handling, edge-case

`player.oldxy2` is only set after `self.timerPos >= 180` (6 seconds at 30 FPS). During the first 6 seconds of gameplay, zombies access `player1.oldxy2` which was set to `self.oldxy` in `gameObject.__init__()` as `(400, 300)`. This isn't a crash, but zombies all converge on center for 6 seconds.

### P1-005: Turret.calcVector crashes when jOn is False
**File:** `gameLib/sprites.py:639-644`
**Confidence:** confirmed
**Agents:** error-handling, logic-correctness

When `jOn == False`, the turret's `self.dir` remains `0` (integer, set on line 621). `calcVector(self.dir)` then tries `dir[0]` and `dir[1]` on an integer, raising `TypeError`. Turrets are unusable in keyboard mode.

### P1-006: No bounds checking on health bar scale
**File:** `gameLib/UI_HUD.py:32`
**Confidence:** confirmed
**Agents:** edge-case, data-integrity

`pygame.transform.scale(self.imageMaster, (self.player.hp, 16))` -- if `hp` reaches 0, this creates a zero-width surface which may cause pygame errors. If hp goes negative before being clamped, it will crash.

### P1-007: Player.fire() with dir=(0,0) creates stationary bullets
**File:** `gameLib/player.py:445-516`
**Confidence:** likely
**Agents:** edge-case, logic-correctness

If the player fires before setting a direction (keyboard mode, no arrow keys pressed), `self.dir = (0, 0)` creates a bullet that doesn't move, sitting on the player indefinitely until `bulletLife` expires.

### P1-008: Shotgun spread calculation uses wrong axes
**File:** `gameLib/player.py:450-503`
**Confidence:** likely
**Agents:** logic-correctness

The shotgun spread logic has multiple issues: `dir1x` and `dir2x` are modified but the conditional branches don't cover all 8 directions. Firing diagonally or in certain directions produces overlapping or incorrectly spread pellets.

### P1-009: Zombie collision with bullet calls on_collide twice
**File:** `gameLib/sprites.py:506-508`
**Confidence:** confirmed
**Agents:** logic-correctness, data-integrity

```python
for collider in collideBul:
    self.on_collide(collider)      # zombie takes damage
    collider.on_collide(collider)  # bullet on_collide called with ITSELF
```

`collider.on_collide(collider)` passes the bullet as its own collider argument. In `Bullet.on_collide`, this means `collider.bulletResist` is 0 (bullet's own resist), so wall resistance is never applied. The hit registration works but the parameter is semantically wrong.

### P1-010: Pickup.on_collide() signature mismatch with subclasses
**File:** `gameLib/pickups.py:28-29`
**Confidence:** confirmed
**Agents:** api-contract, architecture

Base `Pickup.on_collide(self)` takes no player argument, but all subclasses define `on_collide(self, player)`. If base class method is ever called directly, it would fail or behave incorrectly.

### P1-011: Wall collision applies damage to player
**File:** `gameLib/player.py:333-335`
**Confidence:** confirmed
**Agents:** logic-correctness

```python
collideWall = pygame.sprite.spritecollide(self, sGroups.staticSprites, False)
for collider in collideWall:
    self.on_collide(collider)
```

`self.on_collide` subtracts `collider.damage` from player HP. Wall has `self.damage = 0`, so no actual damage occurs, but `godMode` is triggered (`self.hp < hp` is false when damage=0, so actually godMode is NOT triggered). The collision handler still runs unnecessary damage logic for walls.

### P1-012: Flamethrower reload gives only 100 ammo but pickups.py sets 100
**File:** `gameLib/player.py:534`
**Confidence:** confirmed
**Agents:** data-integrity

The reload function gives 100 ammo for the flamethrower, matching the pickup. However, the reload check threshold on line 279 checks `self.ammo < 300`, meaning the player can reload when they have 99 ammo to get back to 100. The threshold should be `< 100`, not `< 300`.

### P1-013: Game over state has no actual game-ending logic
**File:** `gameLib/sprites.py:593-616`
**Confidence:** confirmed
**Agents:** logic-correctness, architecture

When the player dies, the UI shows "Continue? X" countdown and then "GAME OVER", but nothing actually stops the game loop or returns to the menu. The game continues running with a dead player, and zombies keep spawning.

### P1-014: Grenade deceleration is frame-dependent, not time-dependent
**File:** `gameLib/sprites.py:564-570`
**Confidence:** likely
**Agents:** logic-correctness

`self.speedX -= 0.1` per frame means grenade behavior changes if framerate drops. At 30 FPS this takes 40+ frames to stop, but lag spikes would make grenades travel further.

### P1-015: zombieSpawner position randomized every frame
**File:** `gameLib/zombieSpawn.py:20-40`
**Confidence:** confirmed
**Agents:** logic-correctness, performance

`centerRects()` is called every `update()`, re-randomizing the spawner's position within a 10-pixel range each frame. This makes the spawner sprite visually jitter/vibrate.

### P1-016: Level system is disabled but still imported
**File:** `gameLib/main.py:121`
**Confidence:** confirmed
**Agents:** architecture, resource-mgmt

The level loading (`Levels.Level(dataFiles.buildingWallsIm)`) is commented out, but `Levels` is still imported, and the module still runs `Layout()` processing if instantiated. Dead import adds overhead.
